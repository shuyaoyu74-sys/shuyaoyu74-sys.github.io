大模型应用系列

Prompt Tech 提示技术

这个提示技术也可以被破译为提示词工程，是一门在不改变大模型底层权重参数的前提下，通过设计和优化文本，来激发模型能力，引导其输出符号预期结果的技术。

大模型的本质是预测下一个 Token 的概率分布 $P(y|x)$。提示技术的本质，就是通过精心构建上下文 $x$，来降低目标输出 $y$ 的信息熵（不确定性），把模型逼进我们预期的概率空间里。

Prompt 技术有很多分支（如指令格式化、RAG、CoT等），我们率先介绍上下文学习。

上下文学习（ICL）

in-context-learning 上下文学习允许模型在给定的上下文中进行学习和推理，而无需真正更新模型参数，这种方式充分利用模型的预训练知识，并通过在推理阶段提供相关的上下文信息来生成或调整模型输出。

Question
为什么参数固定，模型还能学习？数学原理是啥？

目前学术界主流的解释是“注意力机制即隐式梯度下降”

假设我们在上下文中提供了示例 $s = {(x_1,y_1),(x_2,y_2),…,(x_k,y_k)}$ ，并给出一个新查询$X_test$。在Transformer架构内部，自注意力机制的公式为：

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

当进行前向传播时：

  • 新输入$x_test$被映射为Query ($Q$)
  • 上下文中的示例 $x_i$ 和 $y_i$ 被映射为 Key ($K$) 和 Value ($V$)
  • 通过点积 $\frac{QK^T}{\sqrt{d_k}}$，模型在计算 $x_{test}$ 与之前看到的各个示例 $x_i$ 的相似度。
  • 最终的输出是由这些示例的标签 $y_i$ (对应的 $V$) 加权求和得到的。

更深层次的理论认为，Transformer的前向自注意力计算，在数学形式上等价于在一个局部的，由Prompt构成的“临时数据集”上，进行了一步或多步的线性回归梯度下降。模型利用KVCache中存储的历史动态，动态地构建了一个临时的假设函数$f(x)$，对当前的词元进行预测。

理解了上述底层的数学黑盒过程，我们就能明白：In-Context Learning 能否成功，完全取决于我们在 KVCache 中存入了什么样的历史状态（即上下文中的 $K$ 和 $V$）如果我们给的示例逻辑混乱，模型通过注意力机制计算出的权重就会失效。

因此，回到 Prompt Engineering 的应用层，为了让这个“隐式梯度下降”发挥最大效用，我们在具体操作上，必须通过规范化的字符串拼接，人为地为模型构建出一个清晰、高质量的“临时数据集”。

具体操作就是拼接字符串，我们需要精心构造输入：

  • 操作模板：[任务指令] + [示例1: 输入+输出] + [示例2: 输入+输出] + [新输入: 等待模型补全输出]

那么在拼接之前，当我们手里有一个包含上万条数据的题库时，面临一个问题，那就是我该挑哪几个例子拿出来拼？插一嘴，整个过程发生在推理阶段，而不是训练阶段，整个过程，模型的底层权重参数绝对冻结。

如果每次都靠人工手写或者随机挑选示例，效果会很差，因为随机挑选的例子可能跟当前的问题毫无关系，所以我们需要找一种动态的，自动化的示例检索机制（这其实也是RAG–检索增强生成技术在Prompt构造阶段的早期应用）

ICL示例设计

假设我们现在的任务是让大模型做一个复杂的法律文本分类，我的手里有一个包含10万条已分类的法律文本的本地数据库，现在用户输入一句话：“本合同因不可抗力接触”

第一步，示例挑选，我们要解决谁来教模型这个问题，因为我们不可能把10万条数据都塞进去。

首先要做的就是向量化表示，系统会先调用一个轻量级的编码模型（如 BERT），将用户的 Test Query 转化为一个高维稠密向量 $V_{query}$。同时，你本地的 10 万条历史数据，早就被转化为向量 $V_{doc_i}$ 存放在向量数据库中了。

然后进行相似度计算，计算的核心通常是余弦相似度 或 欧式距离 。以余弦相似度为例，计算 $V_{query}$ 和每一个历史数据向量 $V_{doc_i}$ 的夹角余弦值：

$$\text{Similarity}(V_{query}, V_{doc\_i}) = \frac{V_{query} \cdot V_{doc\_i}}{||V_{query}|| \times ||V_{doc\_i}||}$$

值越接近 1，说明两句话在语义上越相似。

最后，系统根据计算出的相似度得分从高到低排序，挑出得分最高的 $K$ 个例子（就是KNN 算法）。为了避免你选出来的几个例子语义高度重复（模式坍塌），此时会引入多样性计算，比如计算入选示例之间的相互距离，剔除太近的。

经过上述第一步，我们要进行第二步格式融合，也就是prompt组装，来解决怎么教模型这个问题。

现在，算法已经帮你挑出了最相关的 3 个法律文本分类例子。接下来的动作就是纯粹的工程化字符串拼接。

一个高质量的 Prompt 不是随便拼在一块的，它有一个严谨的结构：

  • Task Definition (任务定义)：最开头，你要给模型一个清晰的全局指令。
    • 文本：你是一个专业的法律助手。请判断以下文本属于哪种合同解除类型。
  • Positive Examples (正样本)：把你刚才用 KNN 算出来、选出来的 3 个正确例子，按照特定的分隔符塞进来。
    • 文本：示例1：[甲方违约导致解除] -> 违约解除 \n 示例2：[双方协商一致解除] -> 协议解除
  • Negative Examples (负样本 - 视情况添加)：为了明确边界，有时会人为写入几个容易混淆的错误示范，告诉模型“不要这样做”。
  • Target Query (目标输入)：最后，把用户真正的问题接在末尾，留下悬念，触发模型生成。
    • 文本：现在，请分类以下文本：\n [本合同因不可抗力解除] ->

当然上述只是一部分格式，还有其他的，详见论文：Cross-task generalization via natural language crowdsourcing instructions\

这篇论文提出了一个新的跨任务Instruction数据集。他们使用总包的方式，按照规定的instruction格式，对多个开源数据集进行改造。针对每一个promt，其格式包含：

  • title: 包含一个high-level的任务描述，以及其相关技能，如question generation，answer generation等
  • prompt: 单独的文本命令，一般出现在输入示例之前
  • definition: 指令的补充内容，更加详细地描述指令的具体执行细节
  • things to avoid: 包含模型应该避免的内容，或规则
  • emphasis and caution: 警告或反对的内容
  • positive examples: 提供一个类似系统期望的输入、输出例子，使得众包人员更好地理解任务
  • negative examples: 提供一个类似系统期望的输入、输出的负例，让众包人员尽量避免
  • reason: 解释为什么例子是positive或negative
  • suggestion: 包含一些建议，主要用于指导如何将负例改成正例。

模型在经过上述instruction数据集微调后，能在unseen样本上，达到较好的生成效果。在新样本推断时，将demonstrations加入到task instances即可

第三步，我们进行前向传播，上面组装好的这个巨大且结构严密的字符串，被 Tokenize 之后输入给大语言模型。 接下来的事情就是模型计算出长长的 KVCache，然后利用注意力机制，末尾的 Target Query 通过点积去关注前面精挑细选且排版整齐的 Positive Examples。

输出最终答案：不可抗力解除。

这就是ICL的整个过程。

再扩展一点，第一步示例挑选那里，不只有knn，还有多种其他方法。knn属于启发式方法，它的核心假设是在字面或语义上越相似的句子，作为示例的效果就越好。但这其实存在一个巨大的盲区：人类认为相似的句子，大模型在计算自注意力时未必觉得好用。有时候，稍微有点差异化的例子反而更能激发模型的推理能力。

为了打破这种“只看表面相似度”的局限，学术界引入了进阶方案：LLM-based 方法。这类方法的核心思想是“用魔法打败魔法”——直接利用大模型自身的反馈，来决定到底哪些示例才是好示例。

LLM-based 方法主要有三种技术流派。

第一种基于 Prompt 召回，KNN 用的 BERT 编码器根本不懂大模型喜欢什么， 所以我们要专门训练一个极度懂当前 LLM 口味”的检索器。首先从题库中拿出一个样本作为示例，拼接到输入上喂给 LLM。看 LLM 输出正确答案的概率有多高。概率越高，说明这个示例越“对胃口”。因为不能把题库里几万条数据都过一遍大模型（算力成本太高），所以先用传统的 BM25 或 SBERT 粗筛出一小批候选集。在粗筛的候选集中，把让 LLM 预测概率最高的 $k$ 条作为正样本，概率最低的 $k$ 条作为负样本。然后使用对比学习，分别训练两个编码器：一个是输入编码器（Utterance encoder），一个是示例编码器（Prompt encoder）。目标是让输入向量在连续空间中，无限靠近“好示例”的向量，远离“坏示例”的向量。训练好之后，再来一个新问题，直接用这个专属 Retriever 结合 Faiss（Meta 开源的高效相似度搜索库）进行向量检索。此时找出来的示例，就完全是按照 LLM 的喜好量身定制的了。

第二种是 基于主动学习/强化学习 (RL 序列决策)，有时候即便我们知道哪些单个示例好用，但 ICL 通常需要拼接多个示例（比如 5-shot）。这 5 个示例组合在一起的效果，可不是单个示例效果的简单相加。这引发了指数级的“组合爆炸”问题。所以我们把挑选示例的过程，看作是一个走迷宫或下棋的**马尔可夫决策过程 **，用强化学习（RL）来寻找最优组合。具体做法如下

  • 状态 State ($s_t$)：当前已经选好的示例组合。比如第一步选了 $(x_1, y_1)$，这就是当前状态。
  • 动作 Action ($a_t$)：从庞大的样本库中挑选下一个示例 $(x_2, y_2)$ 加入组合，或者选择输出“停止符号（$\bot$）”表示选够了。
  • 奖励 Reward ($r$)：这是最关键的。当遇到停止符号后，把当前组合的示例拼接成完整的 Prompt，扔进大模型里跑验证集。模型在验证集上的准确率，就是给强化学习算法的 Reward。
  • 训练过程：你可以使用任何经典的 RL 算法（比如类似你在做对齐时可能接触过的策略梯度法），不断试错，最终训练出一个策略网络。这个网络知道在当前已经有什么示例的情况下，下一步抓取哪个示例能让最终组合的 Reward 最大化。

第三种就是基于自生成 (Self-Generation / SG-ICL)，前两种方法再怎么精妙，都极度依赖你手头必须有一个庞大且高质量的外部样本库。如果没有呢？ 没有就让大模型自己凭空捏造出示例，然后再用这些自己生成的示例去解题。第一阶段：系统先用一个极其简单的预设指令，逼迫 LLM 现场写例子。比如，系统输入：Generate a "Negative" review:（生成一个差评：），LLM 根据自身知识权重吐出：it's slow -- very, very slow.，这一步重复几次，凑齐需要的几个 Few-shot 示例。进而将刚才 LLM 自己生成的文本，作为 Context 示例拼接到真正的 Prompt 中。最后附带上用户真实的输入（比如那句很难懂的 movie is so poorly written...）。经过底层 Attention 计算，模型参考着自己刚才生成的示例格式和逻辑，输出了准确的最终预测。

最后，ICL也面临一些挑战，比如大模型对上下文长度有一定限制，处理非常长的上下文可能会受到影响，如何选择适当的上下文是一个关键挑战之一，而且在不同上下文中，模型生成的内容可能会有所不同，确保生成结果的一致性和准确性也是需要关注的问题之一。

那么在标准的上下文学习中，输入和输出的范式是直接映射的：<Input> -> <Output>。 对于情感分类、翻译等浅层模式匹配任务，大模型只需做一次前向传播，通过注意力机制就能轻松算出答案。

但如果你问它一道复杂的数学题，或者一个需要多步推理的逻辑题，直接让它输出最终答案，它大概率会出错。 为什么？因为 Transformer 的层数是固定的。 假设你的模型有 32 层（比如 Llama-2-7B），当它试图直接生成最终答案那个 Token 时，它只能进行 32 次非线性变换。对于极其复杂的逻辑运算来说，这 32 步计算的“深度”根本不足以完成从问题到答案的推导。因此就出现了思维链这种操作。

Chain-of-Thought（CoT）思维链

CoT是一种改进的Prompt技术，目的在于提升大模型在复杂推理任务的表现。这种方式完成了从输入到思维链再到输出的映射，即 ->->

让我们从概率论的角度来看这一动机，在标准的直接问答任务中，输入是问题 $x$，输出是答案 $y$。模型试图直接拟合一个高度复杂且非凸的条件概率分布 $P(y|x)$。对于复杂的逻辑推理题，$x$ 和 $y$ 之间的跨度极大。在预训练语料库中，直接从 $x$ 映射到 $y$ 的数据分布极其稀疏。因此，$P(y|x)$ 的概率地形充满了陡峭的局部最优解，模型极易陷入“幻觉”。CoT 引入了一系列隐变量（Latent Variables） $z = {z_1, z_2, \dots, z_n}$，这些 $z$ 就是中间推理步骤。根据概率论的链式法则，我们将极难拟合的 $P(y|x)$ 进行了分解：

$$P(y, z_1, \dots, z_n | x) = P(z_1|x) \prod_{i=2}^{n} P(z_i | x, z_{因此CoT 将一个“长距离、低概率”的全局跳跃，分解为一系列“短距离、高概率”的局部条件概率连乘**。因为预训练语料（如维基百科、代码、论文）中充满了这种一步接一步的 $P(z_i | z_{i-1})$ 局部推导逻辑，模型拟合这些单步分布的难度呈指数级下降。

再让我们从流行学习的视角来看cot的动机，在深度学习的表征空间中，每一层网络都在对高维流形进行扭曲和拉伸，以实现线性可分。如果强行要求模型直接从 $x$ 的隐空间表征跳跃到 $y$ 的表征，这要求网络映射函数 $f_{\theta}: \mathbb{R}^d \to \mathbb{R}^d$ 具有极端的曲率。这在梯度下降优化中是非常困难的。cot的做法是使中间生成的 Token $z_i$ 重新 Embedding，作为新的输入向量注入到网络中。这就相当于在 $x$ 和 $y$ 之间的高维流形上，打下了一系列“锚点。将一条曲率极高、难以优化的直接映射路径，替换为分段的、局部平滑的线性或低曲率路径（$x \to z_1 \to z_2 \to y$）。

一个完整的cot包括三个部分：指令、逻辑依据、示例。

指令：用于描述问题并且告知大模型的输出格式

逻辑依据：指cot的中间推理过程，可以包含问题的解决方案、中间推理步骤以及问题相关的任何外部知识。

示例：指以少样本的方式为大模型提供输入输出对的基本格式，每一个示例都包含问题、推理过程与答案。

根据是否在 Prompt 中人为提供“示例”，CoT 被严格划分为两大流派：Zero-Shot-CoT 和 Few-Shot-CoT。

先来看Zero-Shot-CoT：它不提供任何人类写好的示例，而是将原始问题，拼接上一句经典的魔法咒语 “Let’s think step by step”，输入给大模型。下面第一张图是生成一个思维链，第二张图则接收来自第一个提示的输出，并从思维链中提取答案，第二个提示是一个自我增强提示。

Question
为什么偏偏是 “Let’s think step by step”？

做实验得来的结果。

Few-Shot-CoT与 Zero-Shot 需要模型“自力更生”写草稿不同，Few-Shot-CoT 是人类直接下场。就是人类在输入真实问题前，先手写几个极度详尽的 [问题 -> 解题步骤 -> 答案] 示例喂给模型。理解了这个两阶段的 Pipeline，你在未来调用 API 开发复杂的 Agent 或写代码时就会明白：为了得到最准确的结果，你可能需要向大模型发起两次 API 请求，第一次要过程，第二次要结果。

当然，cot并不是完美的，在它身上有许多的改进。

改进方法1，CoT-SC: Self Consistency：

论文：《Self-Consistency Improves Chain of Thought Reasoning in Language Models》

主要改进是对答案进行了多数投票，生成多个思路链，取多数答案作为最终答案。

改进方法2，ToT: Tree-of-Thoughts ：

论文：《Tree of Thoughts: Deliberate Problem Solving with Large Language Models》

主要改进是引入多分支探索与启发式搜索（回溯机制），其做法是通过树来建模LLM推理过程，让模型使用不同的思维路径，并提供全新的功能。

改进方法3，GoT: Graph-of-Thoughts (思维图)

论文：《Graph of Thoughts: Solving Elaborate Problems with Large Language Models》

主要核心思想是有向无环图 (DAG)，允许思维的交叉、合并与协同。这里弄明白还是好好去看论文吧。
